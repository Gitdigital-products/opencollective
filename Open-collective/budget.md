Each milestone must include:

1. **Clear deliverables** – What will be produced
2. **Acceptance criteria** – How success is measured
3. **Budget allocation** – Funds released upon completion

5. **Responsible party** – Who is accountable

## Fund Release Triggers

Funds are released when:

| Trigger Type | Description | Approval Required |
|--------------|-------------|-------------------|
| Grant Milestone | Grant recipient completes deliverable | TAB + Grant Manager |
| Operational Milestone | Internal team completes objective | Finance Committee |
| Security Milestone | Audit or bug bounty completed | Security Team |
| Event Milestone | Community event executed | Events Lead |

## Milestone Reporting

Upon completion, each milestone requires:

1. **Completion report** (1-2 pages)
2. **Deliverable artifacts** (code, docs, etc.)
3. **Financial summary** (actual vs. budget)
4. **Lessons learned** (optional but encouraged)

## Milestone Status Definitions

- **Planned**: Defined but not started
- **In Progress**: Actively being worked on
- **On Track**: Progress matches or exceeds timeline
- **At Risk**: Potential delays or issues
- **Blocked**: Cannot proceed without resolution
- **Complete**: All deliverables met
- **Cancelled**: No longer pursuing

## Tracking and Transparency

All milestones are tracked:

- In this document (updated weekly)
- On GitHub Projects board
- In monthly community calls
- In quarterly reports

# GitDigital Grant Structure

**Version 2.0 | Last Updated: March 2025**

## Overview

The GitDigital Grants Program funds development, research, and community initiatives that advance the Aurora ZK Cryptography Framework and the broader GitDigital ecosystem. We believe in supporting contributors who share our vision for transparent, secure, and accessible cryptography.

## Grant Philosophy

- **Open to all** – Anyone can apply regardless of location or background
- **Merit-based** – Funding based on quality, impact, and feasibility
- **Transparent** – All applications, decisions, and outcomes are public
- **Milestone-driven** – Funds released upon verifiable completion
- **Community-governed** – Decisions made by Technical Advisory Board

## Grant Tracks

### Track 1: Core Development

Funding for direct contributions to the Aurora framework.

| Type | Description | Range | Duration |
|------|-------------|-------|----------|
| Feature Development | New cryptographic primitives, protocols | $5k-$50k | 1-6 months |
| Performance Optimization | Speed, memory, or size improvements | $2k-$25k | 1-3 months |
| Security Hardening | Audit preparation, fuzzing, formal verification | $5k-$40k | 1-4 months |
| Infrastructure | CI/CD, testing, deployment tooling | $2k-$20k | 1-2 months |

### Track 2: Research

Funding for exploratory work and novel contributions.

| Type | Description | Range | Duration |
|------|-------------|-------|----------|
| Theoretical Research | New ZK constructions, proofs | $10k-$75k | 3-12 months |
| Applied Research | Implementation studies, benchmarks | $5k-$30k | 2-6 months |
| Literature Review | Comprehensive analysis of specific areas | $2k-$10k | 1-3 months |

### Track 3: Developer Ecosystem

Funding for tools and resources that help developers use Aurora.

| Type | Description | Range | Duration |
|------|-------------|-------|----------|
| Documentation | Tutorials, guides, API references | $2k-$20k | 1-4 months |
| Developer Tools | SDKs, CLIs, debugging tools | $5k-$30k | 1-4 months |
| Example Projects | Demonstrations, templates, starter kits | $1k-$10k | 1-2 months |

### Track 4: Community

Funding for events, education, and community building.

| Type | Description | Range | Duration |

# GitDigital Expense Policy

**Version 2.1 | Last Updated: March 2025**

## Overview

This document outlines the policy for submitting and approving expenses from GitDigital's Open Collective funds. Our goal is to fairly compensate contributors while maintaining transparency and fiscal responsibility.

## Guiding Principles

1. **Fair compensation** – Contributors should be fairly paid for their work
2. **Transparency** – All expenses are publicly visible
3. **Accountability** – Expenses must be legitimate and reasonable
4. **Sustainability** – We spend responsibly to ensure long-term viability
5. **Inclusivity** – Policy applies equally to all contributors

## Who Can Submit Expenses?

- Core Maintainers
- Grant recipients (for approved grants)
- Contributors with approved bounties
- Vendors providing services to GitDigital
- Community members with pre-approved expenses

## Types of Expenses

### Eligible Expenses

| Category | Examples | Documentation Required |
|----------|----------|------------------------|
| **Development work** | Coding, review, testing | GitHub PR/issue links |
| **Documentation** | Writing, editing, translating | Links to published docs |
| **Design** | Graphics, UI/UX, branding | Design files or links |
| **Community** | Event organization, moderation | Event details, screenshots |
| **Travel** | Conference attendance (pre-approved) | Receipts, itinerary |
| **Infrastructure** | Cloud services, domains | Invoices |
| **Software** | Tools, licenses | Receipts |
| **Education** | Courses, materials | Completion certificates |
| **Bounties** | Completed bounty tasks | Bounty issue link |

### Partially Eligible Expenses

| Category | Policy | Approval Required |
|----------|--------|-------------------|
| **Hardware** | 50% of cost, max $1,000 | Core Team |

### Ineligible Expenses

- Personal expenses not related to GitDigital
